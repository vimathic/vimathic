# VIMATHIC — Deployment Guide

## What `npm run build` produces

After `npm run build`, the `dist/` directory contains the runtime app plus a
static documentation site that's indexable by Google and LLM crawlers:

```
dist/
├── index.html                  ~900 KB   ← main app (UI, renderer, three.js, math, recorder)
├── math-worker-XXX.js           ~56 KB   ← Web Worker for off-main-thread math
├── second-screen.html           ~2.4 KB  ← popup page for projector / second monitor
├── vimathic-intro.mp3           ~3.8 MB  ← bundled intro track (S. Melentyev)
├── docs/
│   ├── index.html                7.8 KB  ← documentation landing page (lists all docs)
│   ├── quick-start.html
│   ├── hotkeys.html
│   ├── midi.html
│   ├── camera-programmer.html
│   ├── shader-editor.html
│   ├── recording.html
│   ├── presets.html
│   ├── output.html
│   └── troubleshooting.html
├── sitemap.xml                   ~1 KB   ← XML sitemap for Google Search Console
├── robots.txt                   ~280 B   ← crawler opt-ins (GPTBot, ClaudeBot, etc.)
└── llms.txt                      ~1 KB   ← LLM-discoverability standard
```

**The four runtime files (`index.html`, `math-worker-XXX.js`,
`second-screen.html`, `vimathic-intro.mp3`) must be deployed together.** The `docs/`, `sitemap.xml`,
`robots.txt`, and `llms.txt` files are independent — losing them only affects
discoverability, not the app itself.

> **Note on file size:** `index.html` bundles the entire application including
> Three.js, all 192 formula implementations, the GLSL shader compiler, the GIF
> recorder with its Web Worker, the WebM recorder, post-processing effects,
> and the complete UI — into a single, self-contained file. No CDN requests,
> no lazy loading, no runtime dependencies. The size reflects that it's a
> complete VJ studio, not a landing page.
>
> The static docs site under `dist/docs/` is generated by `plugins/vimathic-docs.js`
> from the same `documents/*.md` files that feed the in-app About modal —
> one source of truth, two outputs (inlined into the bundle for the modal,
> emitted as static `.html` for crawlers).

## Why four runtime files instead of one

VIMATHIC uses `vite-plugin-singlefile` to inline JS, CSS, and three.js into a single `index.html`. Three files cannot be inlined:

1. **`math-worker-XXX.js`** — a Web Worker module. It runs the heaviest math formulas off the main thread to keep 60 fps during complex CPU formulas (Lorenz, FitzHugh-Nagumo, Gray-Scott, 3D Conway, etc.). Module workers cannot be inlined into the main HTML in current Vite, and we chose to keep this file separate rather than rewrite the worker as a classic (non-module) worker.

2. **`second-screen.html`** — the popup page that receives the canvas stream when the user clicks "Second Screen Output". It needs to be a real URL the popup can navigate to.

3. **`vimathic-intro.mp3`** — the bundled intro track. Could be base64-inlined into index.html, but kept separate so the browser can cache it independently of code updates, and stream it during decode instead of loading the entire bundle first. ~3.8 MB.

## What happens if you deploy only `index.html`

The app still works, but with three degradations:

- **MathVisualizer falls back to synchronous main-thread computation.** The console shows `[MathVisualizer] Worker unavailable — math will run synchronously on main thread` once. Frame rate drops noticeably on complex formulas (FitzHugh-Nagumo, Gray-Scott, 3D Conway, anything with > 64² grid). On a fast desktop you may not notice; on mid-range mobile it becomes ~20–30 fps instead of 60.
- **Second-screen popup will fail to open.** The popup tries to navigate to `/second-screen.html`, gets 404, and shows "No stream — reload main window."
- **No intro track on first load.** The fetch of `vimathic-intro.mp3` returns 404; the playlist starts empty. The user has to drag-drop their own audio. A `console.warn` logs `[audio] intro track unavailable`.

The docs site (`dist/docs/`) is independent — if you don't upload it, the in-app About modal still works (its content is inlined into `index.html`), but search engines won't find the documentation as separate pages.

## Deployment checklist

### Static hosting (Cloudflare Pages, Netlify, Vercel, GitHub Pages, S3, CloudFront)

These all deploy the whole `dist/` directory by default. Just point them at `dist/` and you're done. No extra config needed.

The official `vimathic.com` deployment runs on **Cloudflare Pages** — chosen for unlimited bandwidth (important if a single post drives a traffic spike) and built-in privacy-first analytics. Setup is straightforward: connect the Git repo, set build command to `npm run build`, set output directory to `dist`, set `NODE_VERSION=22` in environment variables. No deploy job in CI needed — Cloudflare watches the repo and rebuilds on every push to `main`.

After deployment, submit `sitemap.xml` to Google Search Console:

1. Go to https://search.google.com/search-console
2. Add property → URL prefix → your VIMATHIC URL
3. Verify ownership (DNS TXT record is easiest)
4. Sitemaps → Add new sitemap → enter `sitemap.xml`

Google will crawl your docs within 1–7 days.

### Manual upload (FTP, scp, plain web server)

Copy the entire contents of `dist/` to the server, preserving filenames and directory structure. Verify after upload:

```
your-server.com/index.html
your-server.com/math-worker-XXX.js     ← must be at the same level as index.html
your-server.com/second-screen.html
your-server.com/docs/                  ← directory with the .html docs
your-server.com/sitemap.xml
your-server.com/robots.txt
your-server.com/llms.txt
```

### Verifying the deployment

Open the deployed page in Chrome with DevTools → Network tab open. You should see:

```
✓ index.html         200
✓ math-worker-XXX.js 200    ← if 404, worker fallback is active
✓ (no second-screen request unless you click "Second Screen Output")
```

You can also check by opening the browser console and typing `_vimathic_worker_active` — if `true`, the worker is up and running.

For docs discoverability, check that crawlers can reach the static pages without JS:

```bash
curl -A "Googlebot" https://your-server.com/docs/quick-start.html | head -20
```

You should see the full HTML of the document with `<meta description>` and the article body — no JS execution required.

### Single-file deployment (alternative, if you really need one file)

If your hosting environment can only serve a single file, you can still get full functionality by inlining the worker yourself. There's a known Vite ticket for this: see https://github.com/richardtallent/vite-plugin-singlefile/issues for current status. Until that's resolved, the workflow is:

1. Take `dist/index.html`
2. Take `dist/math-worker-XXX.js`, base64-encode it
3. Replace the worker URL inside `index.html` with `data:application/javascript;base64,<encoded>`

This adds ~75 KB to `index.html` after base64 expansion. The second-screen popup will still require a separate HTML file — there is no workaround for popup navigation.

The static docs site can't be merged into the single file by design — its whole purpose is to be crawlable as separate URLs.

## Performance expectations

With worker enabled (default):
- **Desktop, modern Chrome/Edge:** 60 fps stable on all formulas.
- **Mid-range laptop (integrated GPU):** 60 fps on formulas with grid ≤ 64², 50–60 fps on heavier ones.
- **Mid-range mobile:** 60 fps on simple formulas, 30–40 fps on heavy ones (Gray-Scott, 3D Conway).
- **Old mobile / low-end devices:** graceful degradation via `_throttle = 2` (every other frame), post-processing effects auto-disabled.

Without worker (sync fallback):
- **Desktop:** 60 fps on most formulas, drops to 30–40 fps on the heaviest.
- **Mid-range mobile:** 30 fps on simple formulas, 15–20 fps on heavy.
- **Mobile is essentially unusable** for the new C-tier formulas (FitzHugh-Nagumo, Gray-Scott).

This is why we recommend always shipping the worker file.

## Recording performance notes

The GIF and WebM recorders add additional load during capture:

- **GIF recorder:** uses a separate Web Worker (`gif.js`) for LZW encoding. At 720p × 15fps, capture runs at full speed while encoding happens in the background. Peak memory usage can reach ~1.5 GB for a 60-second capture at maximum settings — the UI warns if estimated usage exceeds 1.5 GB and suggests reducing resolution or duration.
- **WebM recorder:** uses browser-native `MediaRecorder` API. Much lower memory (~50–100 MB for a 60-second capture) but requires Chrome/Edge for VP9 codec support. The composite canvas pipeline (WebGL → 2D overlay with watermark → MediaStream) adds negligible overhead.

Both recorders are designed to not affect the live visual output — capture runs on its own frame schedule independent of the render loop.

## Offline use

Once built, VIMATHIC makes no network requests at runtime — with one exception: a single same-origin fetch of `vimathic-intro.mp3` on first load. After the browser caches it (or the user clicks Clear), no further network activity. Everything else — fonts, shaders, math, UI, recording engine, and the inlined documentation — is bundled into the four runtime `dist/` files. The application works fully offline after the first load. No CDN, no analytics, no telemetry.

The static `dist/docs/` site is also fully offline — each `.html` file is self-contained with inline CSS.
